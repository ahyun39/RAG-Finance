금융법률 질의응답
RAG 시스템 설계·구축·평가

법령 데이터를 직접 수집해 청킹·임베딩·검색·생성까지 엔드투엔드로 만든 RAG 서비스. 만드는 데서 그치지 않고 검색 계층과 생성 계층의 오류를 분리 진단해 개선 지점을 정의했습니다.

01

프로젝트 개요

금융 정보 서비스의 차세대화는 결국 흩어진 도메인 문서를 신뢰 가능한 답변으로 바꾸는 일입니다. 법령처럼 출처와 정확성이 중요한 데이터에서 RAG로 얼마나 신뢰성 있는 답변을 생성할 수 있는지 확인해보고자 했습니다.

문제 정의

  • 법령은 일반인이 검색해도 조문 구조 때문에 이해가 어렵다
  • LLM 단독 답변은 출처 없이 그럴듯한 환각(Hallucination)을 만든다
  • 금융·법률은 "대충 맞는" 답이 곧 리스크가 된다

접근

  • easylaw.go.kr의 생활법령 데이터를 직접 스크래핑·정제
  • 근거 조문을 함께 반환하는 검색 기반 답변 구조
  • 답변 품질을 retrieval / generation으로 분리 측정

03

파이프라인

수집부터 서빙까지 구현한 6단계 흐름. 각 단계가 독립적으로 교체·평가 가능하도록 설계했습니다.

STAGE 1
수집
생활법령 사이트에서 금융 법령·해설을 스크래핑. 제목·본문·주석·표를 유형별로 파싱
Python · requests
STAGE 2
추출
섹션 제목·본문·표를 분리 구조화. 청킹에 적합한 형태로 변환
extract.py
STAGE 3
청킹
조문 단위로 분할. 표는 마크다운으로 별도 청크 생성해 수치 구조 보존
500 / 700자
STAGE 4
임베딩
한국어 문장 임베딩 모델로 벡터화 후 인덱스 구축
ko-sroberta
STAGE 5
검색
코사인 유사도 기반 Top-k 검색으로 근거 조문 후보 추출
FAISS IndexFlatIP
STAGE 6
생성·서빙
검색 결과를 컨텍스트로 LLM이 출처와 함께 답변, 웹 UI로 제공
Upstage Solar API · Streamlit
수집 → law_data.json plv1a/plv2a/plv3a/tplvN 유형별 파싱. tplv\d+ 정규식으로 모든 테이블 variant 수집
추출 → text_sections.json heading · text · note · table 타입 분리. 청킹 전 구조 정제 단계
청킹 → chunks.json 목표 500자 / 최대 700자 / 최소 100자. 표는 마크다운 독립 청크
임베딩 → embeddings.npy jhgan/ko-sroberta-multitask — 한국어 법령 텍스트에 특화된 문장 임베딩
인덱스 → faiss_index.bin IndexFlatIP + faiss.normalize_L2 이중 정규화로 코사인 유사도 정확성 보장
서비스 → app.py Streamlit UI. 유사도 점수 포함 원본 청크 노출. 검색 결과 수 3/5/10 선택
테스트 쿼리 3개("예금자 보호 한도", "보험 청약 취소", "대출 계약 해지") — 코사인 유사도 점수와 함께 상위 청크 반환
# 검색 핵심 — 정규화 임베딩에 내적(IP) = 코사인 유사도 index = faiss.IndexFlatIP(dim) faiss.normalize_L2(doc_vectors) index.add(doc_vectors) q = embed(query); faiss.normalize_L2(q) scores, ids = index.search(q, top_k) # 근거 조문 + 점수 반환
04

주요 기능

🔍

의미 기반 검색

키워드 매칭이 아닌 코사인 유사도(FAISS IndexFlatIP) 기반 벡터 검색. 의미적으로 유사한 법령 청크를 반환합니다.

🤖

AI 답변 생성

Upstage Solar Pro3 LLM에 검색 청크를 컨텍스트로 주입해 문맥 기반 답변을 생성합니다. temperature=0.2로 일관성 확보.

📄

원본 출처 표시

답변 근거가 된 법령 원문 청크를 유사도 점수와 함께 노출해 사용자가 직접 원본을 검증할 수 있습니다.

⚙️

검색 결과 수 조절

3 / 5 / 10건을 선택해 LLM에 전달하는 컨텍스트 청크 수를 유연하게 조정합니다.

평가 — 결과물 판단

이 프로젝트의 핵심은 만든 것이 아니라 왜 틀렸는지를 계층별로 진단한 것입니다. 오답을 검색 계층 문제와 생성 계층 문제로 나누고, 신뢰성 평가 5개 항목에 대해 소스코드 정적 분석으로 판단했습니다.

계층별 오류 진단

계층증상원인개선 방향
RETRIEVAL 관련 조문이 Top-k에 안 들어옴 청크 경계가 조문 맥락을 끊음 / 키워드 매칭 약함 제목 기반 청킹 + 하이브리드 검색(BM25)
RETRIEVAL 맞는 조문이 후순위로 밀림 유사도 점수만으로 랭킹 부족 Cross-Encoder Reranker 도입
GENERATION 검색은 맞는데 답이 어긋남 컨텍스트 근거화(grounding) 부족 프롬프트에 출처 인용 강제
GENERATION 장황하거나 조문 밖 내용 추가 지시 명확성 부족 답변 범위·형식 제약 설계

신뢰성 항목별 판정

평가 항목ㅤㅤ판정내용
Faithfulness (충실성) ⚠ 부분충족 시스템 프롬프트 지침 존재. 사후 자동 검증 레이어 없음. LLM 확률적 동작에 의존
Context Precision (검색 정밀도)ㅤ ⚠ 부분충족ㅤ 유사도 임계값(0.5) 정의됨. LLM 호출 시 results 대신 relevant_results가 전달돼야 하는 버그 식별
Answer Relevance (답변 관련성)ㅤㅤㅤㅤ ⚠ 부분충족 출력 형식·정오답 예시 명시, temperature=0.2 적용. 자동 관련성 측정 없음
수치 정확성 ✕ 미충족 금액·기간 수치 검증 레이어 없음. 금융 도메인에서 수치 오류는 직접적 재정 리스크
Safety (면책 안내) ⚠ 부분충족 UI 고정 문구로 사용자에게 항상 노출. system_prompt 면책 규칙 부재로 LLM 생성 답변 보장 없음
설계 강점 — 출처 투명성(유사도 점수 포함 청크 노출) · 한국어 특화 임베딩 모델 · 표 별도 청킹으로 수치 구조 보존 · 출력 형식 정·오답 예시 제시 · temperature 0.2로 답변 편차 최소화

08

검색 예시

모르는 사람에게 잘못 송금했습니다. 송금한 금액을 돌려받을 수 있는 방법이 있을까요?
착오송금 반환지원제도를 통해 송금한 금액을 돌려받을 수 있습니다. 자금이체 금융회사 등을 통하여 착오 송금한 송금인의 신청이 있는 경우 예금보험공사가 대신 찾아주는 제도입니다. 해당 제도는 「예금자보호법」 제39조의2제1항에 근거하며, 예금보험공사 금융안심포털(fins.kdic.or.kr)에서 신청대상여부 확인 및 반환지원신청을 할 수 있습니다.
금융상품에는 어떤 종류가 있어?
금융상품에는 크게 네 가지 유형이 있습니다. 첫째, 예금성 상품은 예금·예탁금 등 금융기관이 장래에 이자와 함께 돌려주기로 하는 계약입니다. 둘째, 대출성 상품은 대출·신용카드·할부금융 등이 포함됩니다. 셋째, 투자성 상품은 금융투자상품·집합투자증권 등으로 수익을 추구합니다. 넷째, 보장성 상품은 보험·공제 등 위험으로부터 재정적 보장을 제공합니다.
차를 사서 500만원을 12개월 할부로 갚으려고 해. 매달 얼마 내면 돼?
할부로 차를 구매할 경우 매달 상환해야 하는 금액은 대출 금액, 이자율, 상환 기간 등에 따라 결정됩니다. 제공된 참고 자료에는 할부 구매와 관련된 구체적인 상환 금액 계산 방법이 포함되어 있지 않습니다. 정확한 상환 금액을 안내하기 위해서는 해당 금융 상품의 이자율 및 상환 조건을 확인해야 합니다. — 참고 자료 범위 밖 질문에 대한 적절한 비답변 예시

⚠ 이 답변은 참고용 정보이며 법률 자문을 대체할 수 없습니다.
